    Exemples de code (Extraits des notes Obsidian)

    Lors de mon autoformation, j'ai rédigé un cours complet sur la persistance de données en C# avec Entity Framework Core. Voici un exemple concret de configuration de contexte et de mapping relationnel via la Fluent API :

    // Extrait de mes notes Obsidian : Configuration DbContext & Fluent API
using Microsoft.EntityFrameworkCore;

public class KlipDbContext : DbContext
{
    public DbSet<Telechargement> Telechargements { get; set; }
    public DbSet<Fichier> Fichiers { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        base.OnModelCreating(modelBuilder);

        // Configuration Fluent API pour garantir l'intégrité et les index en BDD
        modelBuilder.Entity<Telechargement>(entity =>
        {
            entity.HasKey(e => e.TelechargementId);
            entity.Property(e => e.DateCreation).HasDefaultValueSql("GETUTCDATE()");
            entity.HasIndex(e => e.Reference).IsUnique();
            
            // Relation 1-N avec suppression en cascade
            entity.HasMany(t => t.Fichiers)
                  .WithOne(f => f.Telechargement)
                  .HasForeignKey(f => f.TelechargementId)
                  .OnDelete(DeleteBehavior.Cascade);
        });
    }
}

    Impact algorithmique & Exemples de Code

    Sur Klip, la sensibilisation à l'éco-conception s'est traduite par la mise en avant du calculateur d'empreinte carbone évitée (comparant un envoi éphémère par lien au stockage perpétuel de pièces jointes e-mail dans des datacenters multiples) :

    // Algorithme d'évaluation de l'impact carbone évité (Klip Green IT)
public static double CalculerCarboneEvite(long tailleBytes, int nbDestinataires)
{
    // 1 Mo envoyé par PJ classique = ~19g CO2e (duplication serveurs SMTP + stockage perpétuel)
    double tailleMo = tailleBytes / (1024.0 * 1024.0);
    double impactEmailClassique = tailleMo * 19.0 * nbDestinataires;
    
    // Lien Klip éphémère = stockage unique mutualisé + purge automatique par CRON
    double impactKlip = (tailleMo * 3.5) + (0.5 * nbDestinataires);
    
    return Math.Round(Math.Max(0, impactEmailClassique - impactKlip), 2);
}

    Correctif technique sur la réactivité (Signals)

    Lors des tests de non-régression post-migration, j'ai identifié un bug critique : la mutation directe des tableaux de fichiers lors du glisser-déposer corrompait l'état visuel des Signals en Angular 22. Voici la correction apportée :

    // Correction du bug de réactivité sur les Signals lors du Drag & Drop en Angular 22
import { Component, signal } from '@angular/core';

export class UploadZoneComponent {
  // Signal gérant l'état réactif de la liste des fichiers en cours d'envoi
  files = signal<FileItem[]>([]);

  onFileDropped(newFile: FileItem) {
    // AVANT (Bug) : La mutation directe via .push() n'informe pas le moteur Zoneless
    // this.files().push(newFile);

    // APRÈS (Correction) : Mise à jour immuable (.update) recréant la référence
    this.files.update(currentFiles => [...currentFiles, newFile]);
  }
}

    Ticket 367 — Prolongement légal après signature (+90 jours)

    Lorsqu'un document est signé par un utilisateur, le système prolonge automatiquement sa date d'échéance de 90 jours dans `DAL_Telechargements.cs` pour éviter sa purge prématurée par le CRON :

    // Extrait exact de mes notes (Ticket 367) : DAL/DAL_Telechargements.cs
public static void DesactiverPossibiliteSignature(Guid telechargementId)
{
    using var db = new KlipDbContext();
    var _telechargement = db.Telechargements
        .FirstOrDefault(o => o.TelechargementId == telechargementId);
        
    if (_telechargement != null)
    {
        _telechargement.IsSignable = false;
        
        // Ajout automatique de 90 jours si la conservation n'est pas perpétuelle (HasValue)
        _telechargement.DateEcheance = _telechargement.DateEcheance?.AddDays(90);

        db.Telechargements.Update(_telechargement);
        db.SaveChanges();
    }
}

    Code du PoC de Scellement Cryptographique (Ticket 396)

    Pour hisser Klip au niveau de Signature Avancée (eIDAS), j'ai intégré le service de scellement cryptographique PDFsharp avec horodatage certifié RFC 3161 via l'autorité DigiCert :

    // Extrait de Ticket 396.md : Service de signature inaltérable avec horodatage TSA
using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using PdfSharp.Pdf.Signatures;
using System.Security.Cryptography.X509Certificates;

public static class PdfSignatureService
{
    public static void SignPdf(string inputPdf, string outputPdf, string pfxPath, string pfxPwd, string tsaUrl)
    {
        // 1) Chargement du certificat serveur PFX (avec clé privée exportable)
        var cert = new X509Certificate2(pfxPath, pfxPwd, X509KeyStorageFlags.MachineKeySet | X509KeyStorageFlags.Exportable);

        // 2) Ouverture du document PDF fusionné en modification
        PdfDocument document = PdfReader.Open(inputPdf, PdfDocumentOpenMode.Modify);

        // 3) Configuration de l'autorité d'horodatage certifiée (DigiCert TSA RFC 3161)
        Uri tsaUri = !string.IsNullOrWhiteSpace(tsaUrl) ? new Uri(tsaUrl) : null;
        var signer = new PdfSharpDefaultSigner(cert, PdfMessageDigestType.SHA256, tsaUri);

        // 4) Apposition du scellement cryptographique SHA-256 et sauvegarde
        DigitalSignatureHandler.ForDocument(document, signer, new DigitalSignatureOptions { Reason = "Signature Avancée eIDAS" });
        document.Save(outputPdf);
    }
}
